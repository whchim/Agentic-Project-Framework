# Design Doc: 用户登录

## 1. 基础信息
- Design Doc ID: DD-LOGIN-001
- 关联 Spec: PS-LOGIN-001
- 状态: Draft
- 作者: demo-team
- 创建时间: 2026-05-13

## 2. 设计目标
实现一个安全、简单、可验证的登录流程。

## 3. 总体方案
- 前端提交账号密码
- 后端校验输入
- 查询用户信息
- 验证密码
- 生成 JWT token
- 返回登录结果

## 4. 模块设计
### 4.1 登录接口模块
负责接收登录请求并返回结果。

### 4.2 用户查询模块
负责查询用户是否存在。

### 4.3 密码校验模块
负责比对密码哈希。

### 4.4 Token 生成模块
负责生成 JWT。

### 4.5 风控模块
负责限制错误次数。

## 5. 数据模型
### users 表
- id
- username
- password_hash
- status
- created_at
- updated_at

### login_attempts 表
- id
- username
- attempt_count
- last_attempt_at
- locked_until

## 6. API 设计
### POST /api/login
请求示例：
{
  "username": "demo",
  "password": "123456"
}
成功响应示例：
{
  "success": true,
  "token": "xxx",
  "message": "登录成功"
}
失败响应示例：
{
  "success": false,
  "message": "账号或密码错误"
}

## 7. 核心流程
1.校验输入
2.查用户
3.验证密码
4.检查错误次数
5.生成 token
6.返回结果

## 8. 异常处理
- 用户不存在
- 密码错误
- 账号锁定
- 系统错误

## 9. 安全设计
- 密码使用哈希存储
- token 不写入日志
- 接口加限流
- 防暴力破解

## 10. 性能设计
- 用户查询加索引
- 错误次数判断使用轻量存储
- 登录逻辑避免多余数据库访问

## 11. 可观测性设计
- 记录登录成功/失败次数
- 记录锁定情况
- 记录异常但不输出敏感信息

## 12. 方案对比
- 方案 A：纯账号密码登录
- 方案 B：账号密码 + 验证码
- 当前选择：方案 A，先做最小可用版本
